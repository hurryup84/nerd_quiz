# Nerd Quiz - Implementation Plan

## Overview

A lightweight multiplayer quiz platform for a single community (e.g. a group of friends, colleagues, or a team). This is a single-tenant application — one installation serves one group.

The application revolves around quiz rounds where users answer a single multiple-choice question. Once all required participants have answered, results become visible and a new round can be started.

---

# Goals

* Simple user management
* One active quiz at a time
* Transparent results
* Historical statistics
* Easy question management
* Import/export of question catalog
* Deployable on a simple VPS or web server

# Quiz Rules

## Creating a Quiz

User selects:

* Question
* Required participant count

Example:

```
Question: Q0012
Participants Required: 5
```

The system creates:

```
Status: ACTIVE
```

**Only one quiz may be active at a time.** If a quiz is already active when a user navigates to start a new one:

* The "Start New Quiz" button is disabled
* A message is shown: `A quiz is currently in progress`
* The user is redirected to the active quiz view

---

## Abandoning a Quiz

A quiz can get stuck if not enough participants submit answers. To prevent this:

* Any user can cancel an active quiz
* Cancelling requires confirmation:

```
Cancel this quiz? All submitted answers will be discarded.
[Confirm] [Back]
```

* On confirmation, status changes to:

```
CANCELLED
```

* Cancelled rounds are excluded from statistics
* Cancelled rounds appear in history with a `CANCELLED` badge

Additionally, admins can set an optional timeout when creating a quiz (e.g. 24 hours). When the timeout expires, the round is automatically cancelled.

---

## Answer Submission

Rules:

* One answer only
* Cannot be changed
* Timestamp stored

Database constraint:

```
One answer per user per round
```

---

## Quiz Completion

When enough users have answered:

```
submitted_answers >= required_participants
```

Round status changes to:

```
FINISHED
```

Result visibility becomes enabled.

# Data Model

## Users

```sql
users
-----
id             INTEGER  PRIMARY KEY AUTOINCREMENT
username       TEXT     NOT NULL UNIQUE
password_hash  TEXT     NOT NULL
role           TEXT     NOT NULL DEFAULT 'USER'  -- USER | ADMIN
created_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

INDEX(username)
```

---

## Questions

```sql
questions
---------
id              INTEGER  PRIMARY KEY AUTOINCREMENT
question_id     TEXT     NOT NULL UNIQUE  -- human-readable e.g. Q0001, auto-generated
question_text   TEXT     NOT NULL
answer_a        TEXT     NOT NULL
answer_b        TEXT     NOT NULL
answer_c        TEXT     NOT NULL
answer_d        TEXT     NOT NULL
correct_answer  TEXT     NOT NULL  -- 'A' | 'B' | 'C' | 'D'
created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

INDEX(question_id)
```

Note: `question_id` is auto-generated by the backend (next available Q-number) and is not entered manually by the user.

---

## Quiz Rounds

```sql
quiz_rounds
-----------
id                    INTEGER  PRIMARY KEY AUTOINCREMENT
question_id           INTEGER  NOT NULL REFERENCES questions(id)
required_participants INTEGER  NOT NULL
status                TEXT     NOT NULL DEFAULT 'ACTIVE'  -- ACTIVE | FINISHED | CANCELLED
created_by            INTEGER  NOT NULL REFERENCES users(id)
created_at            DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
finished_at           DATETIME
timeout_at            DATETIME  -- optional, NULL = no timeout

INDEX(status)
```

Statuses:

```text
ACTIVE
FINISHED
CANCELLED
```

---

## Answers

```sql
answers
-------
id               INTEGER  PRIMARY KEY AUTOINCREMENT
quiz_round_id    INTEGER  NOT NULL REFERENCES quiz_rounds(id)
user_id          INTEGER  NOT NULL REFERENCES users(id)
selected_answer  TEXT     NOT NULL  -- 'A' | 'B' | 'C' | 'D'
created_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

UNIQUE(quiz_round_id, user_id)
INDEX(quiz_round_id)
```

Note: Answer submission is rejected if the quiz round status is not `ACTIVE`, preventing race conditions on round completion.

# Question Import / Export

## Format

CSV is used because it is:

* Human editable in any text editor or spreadsheet
* Git friendly (line-based diffs)
* Easy to validate
* Easy to version control
* Widely understood without tooling

---

## CSV Column Order

```
id,question,answer_a,answer_b,answer_c,answer_d,correct_answer
```

`correct_answer` is the letter `A`, `B`, `C`, or `D`.

---

## Example File

```csv
id,question,answer_a,answer_b,answer_c,answer_d,correct_answer
Q0001,Who invented Linux?,Bill Gates,Steve Jobs,Linus Torvalds,Dennis Ritchie,C
Q0002,What does HTTP stand for?,HyperText Transfer Protocol,High Transfer Text Protocol,Hybrid Text Transport Protocol,HyperText Transport Protocol,A
```

---

## Import Rules

Before processing, the backend validates:

* Header row matches expected columns exactly
* Each row has exactly 7 fields
* `correct_answer` is one of `A B C D`
* No blank question text or answers

If validation fails:

```
Import failed: Row 4 — correct_answer must be A, B, C, or D.
```

No rows are imported on validation failure (all-or-nothing).

For each valid row after validation:

### New Question

If `id` does not exist — create the question.

### Existing Questions

If one or more `id` values already exist, the user is asked once before processing:

```
14 questions found. 3 already exist (Q0001, Q0003, Q0007).

How should existing questions be handled?

[Overwrite All]  [Skip All]  [Decide Per Question]
```

"Decide Per Question" shows a simple Yes/No for each conflict.

After import, a summary is shown:

```
Import complete: 11 created, 2 updated, 1 skipped.
```

---

## Export Rules

Export produces a UTF-8 CSV using the exact same column structure.

The file is named:

```
nerd_quiz_export_YYYY-MM-DD.csv
```

Benefits:

* Round-trip compatible
* Directly editable in Excel, LibreOffice, or any text editor
* Easy backup and sharing

# Security

## Authentication

* Password hashing using Argon2
* JWT sessions
* HTTPS only

## JWT Storage

JWT tokens are stored in **httpOnly, Secure, SameSite=Strict cookies** — not in localStorage. This prevents XSS theft of tokens.

Because `SameSite=Strict` is used, CSRF attacks are mitigated by the browser without requiring a separate CSRF token.

Token lifetime:

* Access token: 15 minutes
* Refresh token: 7 days (stored in a separate httpOnly cookie)
* On logout: both cookies are cleared server-side

## Validation

* Input validation on all endpoints using NestJS class-validator
* Rate limiting on login (max 10 attempts per minute per IP)
* All answers validated against enum `A | B | C | D` before persistence

## Rate Limiting

Rate limiting is handled in-process using `@nestjs/throttler`. No Redis dependency needed for MVP. If the platform grows to need distributed rate limiting, Redis can be added later.

## Password Management

Users can change their password from a profile/settings page:

Fields:

* Current password
* New password
* Confirm new password

Validation:

* Current password must match
* New password minimum 8 characters
* Passwords must match

Password reset (forgotten password) is a future enhancement — not in MVP.


Alternative amber mode (toggleable):

```

```

## Typography

* All text: monospace font (`JetBrains Mono`, `Fira Code`, or system `monospace`)
* Headings use uppercase with letter-spacing
* Numbers and stats use a slightly larger weight for emphasis

## UI Components

* Buttons: rectangular, no border-radius, 1px solid border, uppercase label
* Cards: dark surface with 1px green border
* Tables: ASCII-style borders or minimal grid lines in green
* Progress indicators: text-based (e.g. `[=====>    ] 3/5`)
* Correct answer indicator: bright green `[OK]`
* Wrong answer indicator: red `[XX]`
* Cursor blink effect on headings or active elements
* Subtle scanline or noise texture on background (optional, CSS only)

## Example ASCII-style progress bar

```
Participants: [======>   ] 3 / 5
```

## Responsive Design

* Mobile-first layout
* Sidebar navigation collapses to top bar on small screens
* Tables scroll horizontally on mobile

# Database Seeding & First Admin

The backend includes a seed script:

```
cd backend && npx ts-node src/seed/seed.ts
```

The seed script:

* Creates the default admin user (username and password read from environment variables `SEED_ADMIN_USER` and `SEED_ADMIN_PASSWORD`)
* Is idempotent — safe to run multiple times
* Is run once on first deployment

Environment variable example:

```env
SEED_ADMIN_USER=admin
SEED_ADMIN_PASSWORD=change-me-on-first-login
```

# Hosting & Deployment

## Recommended Setup

### Server

* Linux VPS
* 1 vCPU
* 1 GB RAM
* 20 GB SSD

### Services

* Nginx (reverse proxy + static frontend)
* Node.js (NestJS backend via PM2)
* SQLite

### Process Manager

PM2 keeps the Node.js backend running and restarts it on crash:

```bash
pm2 start dist/main.js --name nerd-quiz
pm2 save
pm2 startup
```

### TLS

Certbot (Let's Encrypt) for free HTTPS:

```bash
certbot --nginx -d quiz.example.com
```

### Nginx Config Sketch

```nginx
server {
    listen 443 ssl;
    server_name quiz.example.com;

    # Serve React build
    root /var/www/nerd_quiz/frontend/dist;
    index index.html;
    try_files $uri /index.html;

    # Proxy API
    location /api/ {
        proxy_pass http://localhost:3000;
    }
}
```

### Backup

SQLite database file is backed up nightly:

```bash
# cron: 0 3 * * *
cp /var/www/nerd_quiz/nerd_quiz.db /backups/nerd_quiz_$(date +%F).db
```

### Growth Path

PostgreSQL can replace SQLite using Prisma. The migration is a schema provider swap plus a `prisma migrate deploy`.
